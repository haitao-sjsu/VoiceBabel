## Part 1: Core/ + Services/ 大规模重构 / Core/ + Services/ Large-Scale Refactor

### 1.1 Manager 单一职责化 / Managers Reduced to a Single Responsibility

- `TranscriptionManager` / `TranslationManager` 之前既做"迭代优先级数组调用引擎"，又做"可用性判断"、又做"fallback 状态持有"（`effectiveStartEngine` / `isInFallbackMode`）、又做"语言解析"、又做"超时管理"、又做"后处理"。全部剥掉，只留"迭代 priority 并尝试"这一件事
- `TranscriptionManager` / `TranslationManager` used to (a) iterate the priority array, (b) re-check availability per engine, (c) own fallback state (`effectiveStartEngine` / `isInFallbackMode`), (d) resolve language, (e) manage timeouts, (f) run post-processing. All of that got stripped — the Managers now do only (a)

- 剥干净后的 Manager 拿到空 priority 就立刻抛 `noEngineAvailable`，根本不进入尝试循环。"这次实际用了哪个引擎"改由 Manager 在调用完成后通过 `onTranscriptionResult(text, engine)` / `onTranslationResult(text, engine)` 回调事后上报。调用方不再需要查询 Manager 的内部状态
- Once stripped, the Manager throws `noEngineAvailable` immediately on an empty priority list instead of entering the loop. "Which engine actually ran" is reported post-hoc via `onTranscriptionResult(text, engine)` / `onTranslationResult(text, engine)` callbacks. Callers no longer query Manager internal state

- 这一步即便没有今天的新功能也该做——`effectiveStartEngine` / `isInFallbackMode` 是 `RecordingController` 时代遗留的状态，跟一堆 call-site 耦合，之前每改一次都要提醒自己同步它们。剥干净后 Manager 成为纯粹的无状态组件，可测性、可推理性都提升
- This step was worth doing regardless of today's new feature — `effectiveStartEngine` / `isInFallbackMode` were leftovers from the `RecordingController` era, coupled to a pile of call sites, and every change meant remembering to keep them in sync. Stripped, the Managers are pure stateless components — more testable and easier to reason about

### 1.2 LocalTranslator 协议：抹掉三层可用性护栏 / LocalTranslator Protocol: Erasing Three Layers of Availability Guards

- 之前 `LocalAppleTranslationService` 需要 macOS 15+，结果代码里同时要应付三层护栏：(a) `@available(macOS 15.0, *)` 注解散布在 `AppDelegate` / `AppController` / `TranslationManager` 多处；(b) `#if canImport(Translation)` 条件编译块，5 个文件、13 处；(c) AppDelegate 为了存引用还得用 `Any?` 类型擦除（`var localAppleTranslationService: Any?  // LocalAppleTranslationService, type-erased for availability`）。同一件事被重复表达三次
- `LocalAppleTranslationService` requires macOS 15+, and the codebase was paying for it three times: (a) `@available(macOS 15.0, *)` annotations across `AppDelegate` / `AppController` / `TranslationManager`; (b) `#if canImport(Translation)` conditional-compilation blocks across 5 files at 13 sites; (c) AppDelegate had to type-erase the reference to `Any?` (`var localAppleTranslationService: Any?  // LocalAppleTranslationService, type-erased for availability`). Three layers expressing the same fact

- 新加 `Services/LocalTranslator.swift`：一个没有任何 `@available` 约束的协议（仅一个 async throws 的 `translate` 方法），配一个 `LocalTranslatorFactory.make() -> LocalTranslator?` 工厂。三层护栏统一塌缩到工厂内部的 `if #available(macOS 15.0, *)` 这一行——macOS 14 返回 nil，15+ 返回 live 实例
- Added `Services/LocalTranslator.swift`: a protocol with no `@available` constraint (single async throws `translate` method), plus `LocalTranslatorFactory.make() -> LocalTranslator?`. All three layers collapse into a single `if #available(macOS 15.0, *)` line inside the factory — macOS 14 returns nil, 15+ returns a live instance

- 调用方变化：`AppDelegate` / `TranslationManager` / `AppController` 现在持有一个普通的 `LocalTranslator?`，看不到任何可用性注解。以后要加第二个本地翻译后端（比如另一个离线模型），工厂里多加一行就够
- Call-site impact: `AppDelegate` / `TranslationManager` / `AppController` now hold a plain `LocalTranslator?` with zero availability annotations visible. Adding a second on-device backend later is a one-line change in the factory

- 动机独立于"主观 × 客观"功能——单纯是代码整洁度的债。之前想做几次都因为牵动文件太多搁置，这次借重构的势头一起做了。配套写了 `.claude-tech-research/30_conditional_compilation_translation.md` 论证"为什么用协议比到处撒 `@available` 更好"
- Motivated independently of the "subjective × objective" feature — pure code-hygiene debt. I'd considered doing it before but put it off because it touched too many files; this time I rode the refactor's momentum. Also wrote `.claude-tech-research/30_conditional_compilation_translation.md` to argue "why a protocol beats sprinkling `@available` everywhere"

### 1.3 Service 层：language 变按调用传参，超时与后处理重新归位 / Service Layer: per-call language, timeouts and post-processing moved to their proper homes

- `CloudOpenAIService` / `LocalWhisperService` 之前身上挂着一个可变的 `language: String` 属性，由 AppDelegate 用 Combine sink 与 `settingsStore.whisperLanguage` 同步。这是个潜在 bug 源：两个请求并发时、用户在中间切了 picker，某一个请求会看到"错"的 language
- `CloudOpenAIService` / `LocalWhisperService` used to carry a mutable `language: String` property, synced with `settingsStore.whisperLanguage` by AppDelegate via a Combine sink. Latent bug source: two concurrent requests with the user toggling the picker in between would see the "wrong" language

- 全部改成按调用传参：方法签名加 `language` 参数，由 `TranscriptionManager` 在调用时读 SettingsStore 解析 `"ui"` 哨兵 / auto-detect。Service 成为无状态的"拿参数办事"，彻底消除并发陷阱
- Switched to per-call: method signatures take `language`, and `TranscriptionManager` resolves the `"ui"` sentinel / auto-detect at call time. Services become stateless "take params, do the work" — concurrency trap gone

- 超时内聚进 Service：`CloudOpenAIService` 为 `transcribe` 按音频时长算超时、chat 用固定的 `apiProcessingTimeoutMax`；`LocalWhisperService` 用 `withThrowingTaskGroup` 强制 `min(max(分钟数 × 10, 5s), 90s)`。Manager 不再需要知道这些旋钮
- Timeouts moved inside each Service: `CloudOpenAIService` computes duration-based timeout for `transcribe` and uses `apiProcessingTimeoutMax` for chat; `LocalWhisperService` enforces `min(max(minutes × 10, 5s), 90s)` via `withThrowingTaskGroup`. The Manager no longer needs to know these knobs

- 后处理（标签过滤、trim、繁简转换）从 `LocalWhisperService` 挪到 `TranscriptionManager`，经 `TextPostProcessor`。Service 只负责"音频 → 原始文本"一件事，不再偷偷做文本清理
- Post-processing (tag filtering, trim, Chinese script conversion) moved out of `LocalWhisperService` into `TranscriptionManager` via `TextPostProcessor`. The Service now does only "audio → raw text"; no more covert text cleanup

- （注：`LocalAppleTranslationService.detectSourceLanguage()` 从 `NLLanguageRecognizer` 切到 `NLTagger` 是**周五（04-17）的工作**，只是当天没单独 commit，今天的 `e7d6598` 里一起带上了。详见 `2026-04-17.md`）
- (Note: `LocalAppleTranslationService.detectSourceLanguage()` switching from `NLLanguageRecognizer` to `NLTagger` is **Friday's (04-17) work** — just not committed separately that day; it rode along with today's `e7d6598`. See `2026-04-17.md` for details)

### 1.4 StatusBar 诚实化：预测型图标 → 事后型菜单项 / Status Bar Honesty: Predictive Icon → Post-Hoc Menu Item

- 旧行为：图标随 API mode 变化（Cloud 是 `🎙📶`，Local 是 `🎙🏠`），实质是在预测"下次运行会用的是哪个引擎"——用户一改配置或触发 fallback，这个预测就成了谎言
- Old behavior: icon varied by API mode (`🎙📶` for Cloud, `🎙🏠` for Local), effectively *predicting* "which engine will run next". Any config change or fallback made the prediction a lie

- 新行为：图标只反映 `AppController.AppState`（idle / recording / processing / error），不再掺杂引擎信息。引擎信息改由两条新的信息型菜单项承载——"Last transcription: 🏠 local" / "Last translation: 📶 cloud"——在每次运行**成功之后**由 `setLastTranscriptionEngine` / `setLastTranslationEngine` 填充。事后是诚实的，预测不是
- New behavior: icon reflects only `AppController.AppState` (idle / recording / processing / error) — no engine info mixed in. Engine info is conveyed by two new informational menu items — "Last transcription: 🏠 local" / "Last translation: 📶 cloud" — populated via `setLastTranscriptionEngine` / `setLastTranslationEngine` *after* each successful run. Post-hoc is honest; predictive is not

- `ApiMode` 枚举整个删掉。这个枚举从 Realtime 时代就开始积攒历史债（曾经有 `realtime` 第三态），今天彻底退场
- `ApiMode` enum deleted entirely. It had been carrying historical debt since the Realtime era (there used to be a third case `realtime`); today it retires

## Part 2: 新功能 —— 主观 × 客观引擎可用性 / New Feature: Subjective × Objective Engine Availability

### 2.1 背景：单一 priority 数组表达能力不够 / Motivation: A Single Priority Array Isn't Expressive Enough

- 之前的引擎选择只有一条有序 priority 数组（`transcriptionPriority` / `translationEnginePriority`），UI 只能表达"排序"。有两类用户诉求完全表达不了：
  - "我是 macOS 14，Apple Translation 永远用不了，帮我灰掉并自动跳过"（客观不可用）
  - "我不想把音频上传 OpenAI，即使 API key 是好的"（主观不想用）
- Old engine selection was a single ordered priority array, so the UI could only express "reorder". Two classes of user intent couldn't be expressed:
  - "I'm on macOS 14, Apple Translation will never work — grey it out and skip it" (objective unavailability)
  - "I don't want to send audio to OpenAI, even though the API key works" (subjective preference)

- 今天把这个概念拆成两条正交轴——**主观轴 + 客观轴**——让 UI 能分别表达它们。开工前先写了 `.claude-plan/07_engine_selection_design.md`，把拆分、三种不可用原因、CombineLatest 过滤管道都落到文档里
- Today split the concept into two orthogonal axes — **subjective + objective** — so the UI can express each independently. Before starting, wrote `.claude-plan/07_engine_selection_design.md` documenting the split, the three unavailability reasons, and the CombineLatest filter pipeline

### 2.2 主观轴：SettingsStore 新增 enabled 字典 / Subjective Axis: enabled Dicts in SettingsStore

- 新增 `transcriptionEnabled` / `translationEngineEnabled: [String: Bool]`，由 UI 上每个引擎行的 Toggle 驱动，持久化到 UserDefaults
- Added `transcriptionEnabled` / `translationEngineEnabled: [String: Bool]`, driven by per-engine row Toggles in the UI, persisted to UserDefaults

- init 里对 priority 数组中存在但字典里缺 key 的引擎补齐为 `true`——这让新增引擎时默认启用、老用户升级时 UserDefaults 里没这个 key 也能平滑过渡
- On init, backfill any engine id present in the priority array but missing from the dict to `true` — so new engines default to enabled, and old users upgrading without this key in UserDefaults migrate smoothly

### 2.3 客观轴：EngineAvailabilityProbe / Objective Axis: EngineAvailabilityProbe

- 新增 `Config/EngineAvailability.swift`：一个 `@MainActor` 的 probe 结构体，对每个引擎 id 返回 `.available` 或 `.unavailable(reason)`，reason ∈ `{missingApiKey, osTooOld, localModelNotLoaded}`
- Added `Config/EngineAvailability.swift`: an `@MainActor` probe struct that returns `.available` or `.unavailable(reason)` per engine id, where reason ∈ `{missingApiKey, osTooOld, localModelNotLoaded}`

- 纯函数，不缓存也不持久化——每次调用 `availability(of…:)` 都读当前实时状态。这样就不存在"缓存失效"问题，AppDelegate 在相关信号变化时重新调用即可
- Pure function, not cached, not persisted — every `availability(of…:)` call reads live state. No cache-invalidation problem; AppDelegate just re-invokes on signal change

- 三种 UnavailabilityReason 的选型是为了给 UI 一个明确的 label：UI 不需要自己拼"为什么灰"，只需要 switch 一下 reason
- The three UnavailabilityReason cases exist so the UI has a clear label: the UI doesn't need to construct "why it's greyed" text — it just switches on the reason

### 2.4 AppDelegate 用 CombineLatest 合并四路信号 / AppDelegate CombineLatest Merge

- Manager 拿到的**有效 priority** = `priority ∩ 主观启用 ∩ 客观可用`。这个计算放在 AppDelegate 里，用 `Publishers.CombineLatest` 合并四路信号：priority 数组、enabled 字典、`apiKeyVersion`、WhisperKit 的 `isReadyState`
- The **effective priority** the Manager receives = `priority ∩ subjective-enabled ∩ objectively-available`. AppDelegate computes this via `Publishers.CombineLatest` over four signals: priority array, enabled dict, `apiKeyVersion`, WhisperKit's `isReadyState`

- 任何一路变化都会重新计算、重新推给 `TranscriptionManager.priority` / `TranslationManager.translationEnginePriority`。这正好和 Part 1 §1.1 里"Manager 只迭代、不判断可用性"的职责切分对得上——Manager 信任 AppDelegate 推给它的列表已经过滤好了
- Any of the four changing triggers a recompute and a re-push to `TranscriptionManager.priority` / `TranslationManager.translationEnginePriority`. This dovetails with Part 1 §1.1's "Manager just iterates, doesn't check availability" — the Manager trusts AppDelegate has already filtered the list

- `CombineLatest` 第一次订阅就会用当前值触发一次，替代了之前代码里的"手动初始推送"逻辑——AppDelegate 里少一行 `appController.transcriptionManager.priority = settingsStore.transcriptionPriority` 的初始赋值，避免了"初始推送未过滤 → 过几毫秒被过滤结果覆盖"的瞬态错位
- `CombineLatest` fires on first subscription with current values, replacing the old "manual initial push". AppDelegate loses the `appController.transcriptionManager.priority = settingsStore.transcriptionPriority` initial assignment — avoiding the transient "unfiltered initial push → overwritten by filtered result microseconds later" glitch

### 2.5 SettingsView 行视图与三档徽标 / SettingsView Row View with Three Badges

- Transcription / Translation 两个 section 重写：每个引擎一行，行里有 Toggle（主观启用）、引擎图标、名称/描述或具体的不可用原因（"Requires macOS 15.0" / "API key not configured" / "Model not loaded"），以及三选一的徽标——**Primary / Disabled / Unavailable**
- Rebuilt the Transcription / Translation sections: each engine is a row with a Toggle (subjective enable), icon, name/description or a specific unavailability reason ("Requires macOS 15.0" / "API key not configured" / "Model not loaded"), and one of three badges — **Primary / Disabled / Unavailable**

- Primary 徽标跟的是"第一个既启用又可用的行"——不是总在第 0 行。细节：用户在 macOS 14 上把优先级排成 `[apple, cloud]`，徽标必须落在 `cloud` 上、不能落在 `apple` 上，即使 `apple` 在数组里更靠前
- Primary tracks "first enabled-and-available row" — not always row 0. Subtlety: on macOS 14 with priority `[apple, cloud]`, the badge must land on `cloud`, not `apple`, even though `apple` is earlier in the array

- Section 底部的说明文字在有效列表为空（所有引擎要么被用户关掉、要么客观不可用）时变红，把"你把自己堵死了"这种状态在 UI 上暴露，而不是等调用时抛错才发现
- The section footer turns red when the effective list is empty (every engine either disabled or unavailable), surfacing the "you've painted yourself into a corner" state in the UI rather than at call time

- `SettingsWindowController` 现在把 `EngineAvailabilityProbe` 注入到视图，所以用户正在编辑 API key 或 WhisperKit 后台加载完成时，每一行都能实时重判可用性
- `SettingsWindowController` now injects `EngineAvailabilityProbe` into the view, so rows re-evaluate availability live as the user edits the API key or WhisperKit finishes loading

### 2.6 顺带删掉的 EngineeringOptions.enableModeFallback / Dropped EngineeringOptions.enableModeFallback

- 这个开关的语义在新模型里自动吸收了："禁用 fallback" 现在等同于"把其他引擎的主观开关关掉"或"把 priority 数组改成只有一个元素"。两套机制表达同一件事是冗余，删掉更干净
- This flag's semantics got absorbed by the new model: "disable fallback" now equals "turn off the subjective toggle on other engines" or "reduce the priority array to one element". Two mechanisms expressing the same thing is redundant; removing is cleaner
