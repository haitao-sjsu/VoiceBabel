# 2026-04-20 Development Log / 开发日志

### 1. Engine Selection Redesign: Subjective × Objective Split / 引擎可选性重新设计：主观 × 客观两条轴

- Before today, "which engine to use" was a single ordered priority array (`transcriptionPriority` / `translationEnginePriority`) and the Managers figured out availability at call time. Problem: the UI could only express "reorder", not "I have macOS 14, Apple Translation will never work — please grey it out and skip it", nor "I don't want to use Cloud for privacy reasons, even though the API key works"
- 今天之前，"用哪个引擎"只有一条有序 priority 数组（`transcriptionPriority` / `translationEnginePriority`），可用性是 Manager 在调用时现判。问题：UI 只能表达"排序"，没法表达"我是 macOS 14，Apple Translation 永远用不了，帮我灰掉并跳过"，也没法表达"出于隐私我不想用 Cloud，即使 API key 是好的"

- Split the concept into two orthogonal axes:
  - **Subjective** (user preference): new `transcriptionEnabled` / `translationEngineEnabled: [String: Bool]` in SettingsStore, driven by UI toggles, persisted in UserDefaults
  - **Objective** (system reality): new `Config/EngineAvailability.swift` — `@MainActor` struct `EngineAvailabilityProbe` that returns `.available` or `.unavailable(reason)` per engine id, where reason ∈ `{missingApiKey, osTooOld, localModelNotLoaded}`. Pure function, not cached, not persisted
- 把这一个概念拆成两条正交轴：
  - **主观**（用户偏好）：SettingsStore 新增 `transcriptionEnabled` / `translationEngineEnabled: [String: Bool]`，由 UI 开关驱动，持久化到 UserDefaults
  - **客观**（系统实情）：新增 `Config/EngineAvailability.swift`——`@MainActor` 的 `EngineAvailabilityProbe` 结构体，对每个引擎 id 返回 `.available` 或 `.unavailable(reason)`，reason ∈ `{missingApiKey, osTooOld, localModelNotLoaded}`。纯函数，不缓存也不持久化

- The effective priority that Managers receive = `priority ∩ subjective-enabled ∩ objectively-available`. AppDelegate computes this via `Publishers.CombineLatest` over the four relevant signals (priority, enabled-dict, apiKeyVersion, WhisperKit `isReadyState`) and pushes the filtered list into `TranscriptionManager.priority` / `TranslationManager.translationEnginePriority` on every change
- Manager 拿到的有效 priority = `priority ∩ 主观启用 ∩ 客观可用`。AppDelegate 用 `Publishers.CombineLatest` 合并四路信号（priority、启用字典、apiKeyVersion、WhisperKit 的 `isReadyState`）后计算这个交集，任何一路变化都重新计算并推到 `TranscriptionManager.priority` / `TranslationManager.translationEnginePriority`

- Managers simplify correspondingly: they no longer re-check availability per engine — they iterate the list as-is. If the list is empty they throw `noEngineAvailable` immediately instead of attempting and failing. `effectiveStartEngine` / `isInFallbackMode` state removed from TranscriptionManager; "which engine actually ran" is now reported post-hoc via `onTranscriptionResult(text, engine)` callback
- Manager 相应简化：不再逐个引擎再判一次可用性，直接迭代拿到的列表。列表为空就立刻抛 `noEngineAvailable`，不再试了才失败。TranscriptionManager 的 `effectiveStartEngine` / `isInFallbackMode` 状态删除；"实际跑的是哪个引擎"改由 `onTranscriptionResult(text, engine)` 回调事后上报

- Along the way, `EngineeringOptions.enableModeFallback` was dropped — "禁用 fallback" now means "把其他引擎的 subjective enable 关掉，或让 priority 只保留一个"，语义更清晰
- 顺带删掉了 `EngineeringOptions.enableModeFallback`——"禁用 fallback"现在就是"把其他引擎的主观开关关掉，或让 priority 只保留一个"，语义更干净

### 2. LocalTranslator Protocol: Erase the macOS 15 Availability Gate / 用协议抹掉 macOS 15 可用性护栏

- `LocalAppleTranslationService` requires macOS 15.0+, which was polluting storage / call sites with `@available(macOS 15.0, *)` annotations, `#if canImport(Translation)` blocks, and `Any?` type erasure in AppDelegate (`var localAppleTranslationService: Any?  // LocalAppleTranslationService, type-erased for availability`). Ugly
- `LocalAppleTranslationService` 需要 macOS 15.0+，结果 `@available(macOS 15.0, *)` 注解、`#if canImport(Translation)` 块、AppDelegate 里 `Any?` 类型擦除 (`var localAppleTranslationService: Any?  // LocalAppleTranslationService, type-erased for availability`) 散落到各处。丑

- Introduced `Services/LocalTranslator.swift`: a protocol with a single `translate(text:sourceLanguage:targetLanguage:) async throws -> String` method (no availability constraint on the protocol itself), plus `LocalTranslatorFactory.make() -> LocalTranslator?` that encapsulates the `if #available(macOS 15.0, *)` gate. On macOS 14 the factory returns `nil`; on 15+ it returns a live `LocalAppleTranslationService`
- 新增 `Services/LocalTranslator.swift`：协议本身不带可用性约束，只有一个 `translate(text:sourceLanguage:targetLanguage:) async throws -> String` 方法；配套 `LocalTranslatorFactory.make() -> LocalTranslator?` 把 `if #available(macOS 15.0, *)` 这一个决策封闭在工厂里。macOS 14 返回 `nil`，15+ 返回 `LocalAppleTranslationService` 实例

- Net effect: `AppDelegate` / `TranslationManager` / `AppController` now hold a plain `LocalTranslator?` and never see availability annotations. Adding a second on-device translation backend in the future is a one-line change inside the factory
- 净效果：`AppDelegate` / `TranslationManager` / `AppController` 现在持有普通的 `LocalTranslator?`，再看不到可用性注解。以后要加第二个本地翻译后端，只需在工厂里多加一行

- Wrote `.claude-tech-research/30_conditional_compilation_translation.md` to record the reasoning (why not just ship two binaries, why a protocol wins over `@available` sprinkled everywhere) so future-me doesn't re-litigate this
- 写了 `.claude-tech-research/30_conditional_compilation_translation.md` 记录思考过程（为什么不直接出两套二进制、为什么协议比到处撒 `@available` 更好），省得以后再纠结一遍

### 3. Service Layer: Language Becomes a Per-Call Parameter / Service 层：语言变成按调用传参

- Previously `CloudOpenAIService` / `LocalWhisperService` carried a mutable `language: String` property that AppDelegate kept synced to `settingsStore.whisperLanguage` via a Combine sink. This was a latent bug source: two requests racing could see the "wrong" language if the user toggled the picker between them
- 之前 `CloudOpenAIService` / `LocalWhisperService` 身上挂着一个可变的 `language: String` 属性，AppDelegate 用 Combine sink 把它和 `settingsStore.whisperLanguage` 同步。这是个潜在 bug 源：两个请求并发时，如果用户在中间切换 picker，某一个请求可能看到"错"的 language

- Moved language resolution into `TranscriptionManager` (who reads `SettingsStore` and resolves the `"ui"` sentinel / auto-detect at call time), and made the Services' methods accept `language` as a per-call parameter. AppDelegate's sink on `$whisperLanguage` now only logs the change — no more cross-service mutation
- 把语言解析上移到 `TranscriptionManager`（由它在调用时读 `SettingsStore` 并解析 `"ui"` 哨兵 / auto-detect），Service 的方法签名改为按调用接收 `language` 参数。AppDelegate 订阅 `$whisperLanguage` 的 sink 现在只打日志，不再跨 Service 写值

- Timeouts likewise moved inside each service: `CloudOpenAIService` computes duration-based timeout for `transcribe` and uses `apiProcessingTimeoutMax` for chat; `LocalWhisperService` enforces `min(max(minutes × 10, 5s), 90s)` via `withThrowingTaskGroup`. The Manager doesn't need to know these knobs
- 超时也挪到 Service 内部：`CloudOpenAIService` 为 `transcribe` 按音频时长算超时、chat 用 `apiProcessingTimeoutMax`；`LocalWhisperService` 用 `withThrowingTaskGroup` 强制 `min(max(分钟数 × 10, 5s), 90s)`。Manager 不需要关心这些旋钮

- Post-processing (tag filtering, trim, 繁简转换) moved out of `LocalWhisperService` into `TranscriptionManager` via `TextPostProcessor`. Service returns raw joined text — a single service is no longer two things (transcription + text cleanup)
- 后处理（标签过滤、trim、繁简转换）从 `LocalWhisperService` 挪到 `TranscriptionManager`，经 `TextPostProcessor`。Service 返回原始拼接后的文本——一个 Service 不再同时干两件事（转录 + 文本清理）

### 4. Status Bar: Icon Reflects State, Menu Reflects Engine / 状态栏：图标只看状态，菜单看引擎

- Old behavior: icon varied by API mode (`🎙📶` for Cloud, `🎙🏠` for Local), which meant the icon was implicitly asserting "this is what *will* run next" — a lie when the user's config changed or fallback fired. Also carried a three-way `ApiMode` enum (local/cloud/realtime-before-it-was-removed) that kept getting stale
- 旧行为：图标随 API mode 变化（Cloud 是 `🎙📶`，Local 是 `🎙🏠`），等于在暗示"下次运行的会是这个"——可一旦用户改配置或触发 fallback 就变成谎话。还携带一个三态的 `ApiMode` 枚举（local/cloud/realtime-before-it-was-removed），老是滞后

- New behavior: icon reflects only `AppController.AppState` (idle/recording/processing/error). Engine information moved to two new informational menu items — "Last transcription: 🏠 local" / "Last translation: 📶 cloud" — populated post-hoc via `setLastTranscriptionEngine` / `setLastTranslationEngine` after each successful run. Post-hoc is honest; predictive was not
- 新行为：图标只反映 `AppController.AppState`（idle/recording/processing/error）。引擎信息挪到两条新的信息型菜单项——"上次转录：🏠 local" / "上次翻译：📶 cloud"——由 `setLastTranscriptionEngine` / `setLastTranslationEngine` 在每次运行成功后事后填充。事后是诚实的；预测的不是

- Deleted `ApiMode` enum entirely. Also killed the unwired NetworkHealthMonitor callbacks in AppDelegate — `recoverFromFallback` / `isInFallbackMode` no longer exist after the refactor. `NetworkHealthMonitor` the file survives but is dead code, flagged for a follow-up PR
- `ApiMode` 枚举整个删掉。同时砍掉 AppDelegate 里没再连通的 NetworkHealthMonitor 回调——`recoverFromFallback` / `isInFallbackMode` 在重构后已经不存在。`NetworkHealthMonitor.swift` 文件还在但是死代码，留给后续 PR 清理

### 5. SettingsView: Rows with Toggles, Badges, and Live Unavailability / SettingsView：带开关、徽标和实时不可用提示的引擎行

- Rebuilt the Transcription / Translation sections: each engine renders as a row with a Toggle (subjective enable), the engine icon, either its name/description or a specific unavailability reason ("Requires macOS 15.0" / "API key not configured" / "Model not loaded"), and exactly one of three badges — **Primary / Disabled / Unavailable**
- 重写了 Transcription / Translation 两个 section：每个引擎渲染为一行，包含 Toggle（主观启用）、引擎图标、名称/描述或具体的不可用原因（"需要 macOS 15.0" / "未配置 API key" / "模型未加载"），以及三选一的徽标——**Primary / Disabled / Unavailable**

- The Primary badge tracks "first row that is both enabled and available" — not always row 0. This is a subtle correctness point: if the user reordered `[apple, cloud]` on macOS 14, the badge has to sit on `cloud`, not `apple`, even though `apple` is higher in the priority array
- Primary 徽标跟的是"第一个既启用又可用的行"——不一定是第 0 行。这里有个细微的正确性点：用户在 macOS 14 上把优先级排成 `[apple, cloud]`，徽标必须落在 `cloud` 上、不能落在 `apple` 上，即使 `apple` 在 priority 数组里更靠前

- Section footer turns red when the effective list is empty (every engine either disabled or unavailable), surfacing the "you've painted yourself into a corner" state at the UI boundary instead of at call time
- 当有效列表为空（所有引擎要么被用户关掉、要么客观不可用）时，section 底部的说明文字变红，在 UI 边界而非调用时就把"你把自己堵死了"这种状态暴露出来

- `SettingsWindowController` now injects the `EngineAvailabilityProbe` into the view so rows can re-evaluate availability live as the user edits the API key or WhisperKit finishes loading in the background
- `SettingsWindowController` 现在把 `EngineAvailabilityProbe` 注入视图，这样用户正在编辑 API key 或 WhisperKit 后台加载完成时，每一行都能实时重判可用性
