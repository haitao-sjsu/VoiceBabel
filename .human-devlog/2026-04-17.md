### 1. Bug Triage: Apple Translation Rejects Mixed Zh/En Speech / 定位 Apple Translation 拒绝中英混杂语音的 Bug

- 做了一次中英混杂的语音翻译，转录结果是 "为什么在初始化的时候只需要初始化那个 cloud open ai service，不需要初始化那个 local apple translation service?"。这句话里汉字占约 35%，但翻译直接失败，日志显示 Apple Translation 被跳过、回退到 Cloud
- Did a mixed Chinese-English voice translation. The transcription was "为什么在初始化的时候只需要初始化那个 cloud open ai service，不需要初始化那个 local apple translation service?". About 35% of the characters are Chinese, but the translation failed outright — the log showed Apple Translation was skipped and fell back to Cloud

- 根因定位：`LocalAppleTranslationService.detectSourceLanguage()` 用 `NLLanguageRecognizer.dominantLanguage` 把这句话识别成了英文，于是 pre-check 判定是 en→en（目标语言也是英文），被判为 unsupported pair 直接拒绝，根本没进入翻译步骤
- Root cause: `LocalAppleTranslationService.detectSourceLanguage()` uses `NLLanguageRecognizer.dominantLanguage`, which classified this sentence as English. The pre-check then saw en→en (target also English), ruled it an unsupported pair, and rejected the call without ever entering the translation step

- Wrote a quick test to verify `NLLanguageRecognizer`'s actual behavior across 5 representative inputs. The results were striking:

| # | 文本 / Text | dominantLanguage | Top hypotheses |
|---|------|-----------------|----------------|
| 1 | 为什么在初始化的时候只需要初始化那个 cloud open ai service，不需要初始化那个 local apple translation service? | **en** | en: 0.5773, da: 0.0589, nl: 0.0501, pt: 0.0462, nb: 0.0342 |

- Case 1 is the problem: 35% Chinese characters, and Chinese doesn't appear in the top-5 hypotheses *at all*. Tried `languageHints` and `languageConstraints` — neither helped, some made it worse. Pure Chinese (cases 3, 5) and pure English (case 4) work fine. **NLLanguageRecognizer's statistical model fails systematically on mixed Chinese-English input** — this is a blind spot in how the model represents language, not a probability issue

### 3. Architectural Question: Why Are We Pre-Detecting at All? / 架构反思：我们为什么要自己先检测？

- 用户提了个好问题：Apple Translation 不是支持 `source: nil` 自动检测吗？我们为什么要自己先跑一遍 NLLanguageRecognizer？
- User raised a sharp question: doesn't Apple Translation support `source: nil` auto-detect? Why are we running NLLanguageRecognizer ourselves first?

- 翻代码注释重新确认了当时的理由（`LocalAppleTranslationService.swift` 第 88 行）：当 `TranslationSession` 在 `source: nil` + 缺语言包的情况下，**不抛错而是挂起（hang）**。所以我们先手动检测、再用 `LanguageAvailability.status()` 预检查语言包是否安装，未安装就快速失败 + fallback 到 Cloud
- Re-read the code comment at `LocalAppleTranslationService.swift:88` and confirmed the original reason: when `TranslationSession` runs with `source: nil` + missing language pack, it **hangs instead of throwing**. So we manually detect first, then `LanguageAvailability.status()` pre-checks the pack, and fail-fast + fall back to Cloud if missing

- 这个设计本身是合理的——它规避了 hang 的坑。但它把"语言检测的正确性"这个包袱转嫁到了 NLLanguageRecognizer 上，而我们今天发现 NLLanguageRecognizer 扛不住这个责任。所以不是要撤掉 pre-detection，而是要换一个更靠谱的检测器
- The design is sound on its own terms — it sidesteps the hang. But it transfers the "correct language detection" burden onto NLLanguageRecognizer, and today proved NLLanguageRecognizer can't carry it. So the fix isn't to remove pre-detection; it's to replace the detector with something more reliable

- 顺带回应了"能不能用大模型做语言检测"：可以，但对这个场景是杀鸡用牛刀——Apple Translation 路径本来是想走本地离线的，再走一次 API 等于把离线优势抵消了。Cloud 翻译路径本身不存在这个问题（GPT 自己能处理混杂）
- Also considered "can we use an LLM for language detection": yes but overkill for this case — the whole point of the Apple Translation path is offline/local, and routing through an API would negate that. The Cloud translation path doesn't have this problem anyway (GPT handles mixed input natively)

### 4. Alternative: NLTagger Word-Level Tagging / 替代方案：NLTagger 逐词标注

- `NLTagger` 是 `NaturalLanguage` 框架里另一个类，可以按词级标注语言——而不是对整段文本给一个全局判定。实测对所有 5 个案例都正确：案例 1 里的中文词被标为 zh，英文词被标为 en，只要按字符数加总就能得出"中文占多数"
- `NLTagger` is another class in the `NaturalLanguage` framework that tags languages at *word* level rather than emitting one global verdict for the whole string. Tested against all 5 cases — all correct: in case 1, Chinese words are tagged zh, English words tagged en, and summing by character count gives "Chinese majority"

- 一个 edge case：极短纯英文文本 "Hello World" 被 NLTagger 识别为意大利语。但这只影响极短纯英文输入，而纯英文这一档 `NLLanguageRecognizer` 本来就不会出错——所以新方案在它擅长的地方超过旧方案，在旧方案本来就对的地方没问题。净收益
- One edge case: the very short pure-English string "Hello World" gets tagged as Italian by NLTagger. But this only affects extremely short pure-English input, and NLLanguageRecognizer had no trouble on pure English to begin with — so the new approach wins where the old one fails, and ties where the old one succeeded. Net win

### 5. Implementation: Switch detectSourceLanguage to NLTagger / 改造 `detectSourceLanguage` 切换到 NLTagger

- 在 `LocalAppleTranslationService.detectSourceLanguage()` 里用 `NLTagger` 按词标注 + 字符数加权统计替换 `NLLanguageRecognizer.dominantLanguage`。当存在非英文 token 时，优先选非英文的最高频语言——这符合产品直觉"中英混杂倾向于当作中文处理"，也符合这个函数存在的初衷（Apple Translation 本身能处理英文，挂起风险主要来自小语种的缺包）
- Replaced `NLLanguageRecognizer.dominantLanguage` in `LocalAppleTranslationService.detectSourceLanguage()` with NLTagger word-level tagging + character-count weighting. When non-English tokens are present, prefer the highest-frequency non-English language — matches the product intuition ("mixed zh/en should lean toward Chinese") and matches this function's original purpose (Apple Translation handles English natively; the hang risk comes from minority languages with missing packs)
